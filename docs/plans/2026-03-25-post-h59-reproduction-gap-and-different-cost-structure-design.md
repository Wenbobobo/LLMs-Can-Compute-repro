# 2026-03-25 Post-H59 Reproduction Gap And Different-Cost-Structure Design

## Purpose

This design converts the landed `H58` closeout into a cleaner next-stage
control surface:

- preserve the honest narrow positive result;
- preserve the landed value-negative stop on the current executor lane;
- state clearly how far the repository still is from the broader Percepta
  framing;
- prepare one self-contained GPTPro re-interview dossier; and
- constrain any future reopen to materially different cost structures only.

This is a consolidation-and-routing wave, not a new runtime execution wave.

## Current Scientific Reading

The current branch supports three claims simultaneously:

1. append-only trace execution plus exact latest-write retrieval is real on the
   admitted narrow surfaces;
2. bounded useful-case execution on that substrate is real enough to preserve
   `H43` as a paper-grade endpoint; and
3. bounded executor/system value on the best currently justified route is now
   disconfirmed, including the strongest current native useful-kernel test
   `R62`.

The right live interpretation is therefore:

- mechanism support survives;
- the broader executor-value lane is closed;
- the project is not globally dead; but
- the current lane is dead enough that repeating it by momentum is not
  scientifically justified.

## Admissible Future Reopen Rule

Any later scientific reopen must satisfy all of the following before runtime
execution starts:

1. the new route changes the cost structure materially rather than merely
   changing syntax or surface coverage;
2. the new route targets an admitted useful case or a clearly stronger narrow
   benchmark, not a flashy broader demo;
3. stop rules are written before execution;
4. the work runs from a clean successor worktree under
   `D:/zWenbo/AI/wt/...`; and
5. the route explains why the new comparator is better than repeating the
   already-falsified `R62` lane.

If these conditions are not met, the scientifically correct action is to stop
or remain in planning-only mode.

## Candidate Families For F33

`F33` stores planning-only candidate families, not authorizations.

The top current families are:

1. an amortized or indexed trace-retrieval route where repeated queries share
   enough structure that the present per-step Python retrieval overhead is no
   longer the dominant cost;
2. a low-level compiled or external coprocessor route where retrieval kernels
   are implemented under a materially different runtime contract from the
   current interpreter-heavy path; and
3. a transformed or trained executor route only if it changes the cost model
   and still targets the useful-case endpoint honestly rather than escaping the
   exact lane rhetorically.

`F27`, `R53`, and `R54` remain blocked by default. `F33` stores guarded future
options, not permission.

## GPTPro Dossier Rule

The GPTPro re-interview must be treated as a precious one-shot consulting pass.

That means the saved dossier must:

- be self-contained because GPTPro cannot browse the repo;
- state the scientific target, current evidence stack, and exact negative
  results explicitly;
- explain the distance from the current branch to the public Percepta
  headline;
- include a small set of key code excerpts rather than file-path-only
  references; and
- ask for one integrated judgment, not many short loosely coupled prompts.
